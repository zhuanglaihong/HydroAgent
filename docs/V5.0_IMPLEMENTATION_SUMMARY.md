#### 3.1 超时保护
```python
def _execute_with_timeout(self, func, timeout_seconds, *args, **kwargs):
    """
    使用 signal.alarm 实现超时保护
    - Unix: signal.SIGALRM
    - Windows: 优雅降级（无超时保护）
    """
    # 设置超时信号
    signal.signal(signal.SIGALRM, timeout_handler)
    signal.alarm(timeout_seconds)

    try:
        result = func(*args, **kwargs)
        return {"success": True, "result": result}
    except TimeoutError:
        return {"success": False, "error_type": "timeout", "retryable": True}
    finally:
        signal.alarm(0)  # 取消超时
```

#### 3.2 自动重试
```python
def _execute_with_retry(self, func, *args, **kwargs):
    """
    自动重试机制（指数退避）
    - max_retries: 3次
    - retry_backoff: 2^n 秒
    """
    retry_count = 0
    while retry_count < self.max_retries:
        result = self._execute_with_timeout(func, self.timeout, *args, **kwargs)

        if result["success"]:
            return result

        if not result.get("retryable", False):
            return result  # 不可重试，直接返回

        # 指数退避
        wait_time = self.retry_backoff ** retry_count
        time.sleep(wait_time)
        retry_count += 1

    return {"success": False, "error_type": "max_retries_exhausted"}
```

## 📊 架构对比

### v3.5 vs v5.0

| 特性 | v3.5 (旧) | v5.0 (新) |
|------|-----------|-----------|
| **执行模型** | 固定5阶段流水线 | 状态机驱动循环 |
| **任务类型** | 8种硬编码 | 无限（LLM生成）|
| **错误处理** | 失败即终止 | 自动重试+恢复 |
| **反馈流** | 单向（顺序） | 双向（路由） |
| **优化方式** | 手动迭代 | 自动（目标驱动）|
| **案例复用** | 规则匹配 | FAISS语义搜索 |

### 状态机核心流程

```
INIT → WAITING_USER_INPUT → ANALYZING_INTENT → INTENT_CONFIRMED
  ↓
PLANNING_TASKS → PLAN_READY
  ↓
GENERATING_CONFIG ←─┐ (重试<3次)
  ↓                 │
CONFIG_READY        │
  ↓                 │
EXECUTING_TASK ←────┘ (重试<3次)
  ↓
EXECUTION_SUCCESS → ANALYZING_RESULTS
  ↓
RESULTS_ACCEPTABLE → COMPLETED_SUCCESS
  OR
RESULTS_BELOW_THRESHOLD → ITERATING → 回到PLANNING_TASKS
  (迭代优化)
```

## 📝 已知问题和限制

### 1. Windows超时限制

**问题**: `signal.SIGALRM` 仅在Unix系统可用

**影响**: Windows上无法强制超时终止

**缓解**:
- 使用 `threading.Timer` 作为替代（已考虑但未实现）
- 当前方案：依赖hydromodel自身的超时机制

### 2. PromptPool性能

**问题**: FAISS索引加载可能较慢（首次）

**影响**: 首次执行略慢（~500ms）

**缓解**:
- 使用持久化索引
- 异步加载（future）

### 3. 测试覆盖不完整

**问题**: RunnerAgent没有专门的重试测试
