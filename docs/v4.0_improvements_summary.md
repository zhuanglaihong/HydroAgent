## 🎯 核心改进

### 1. 代码生成文件统一管理

**问题**：之前生成的代码保存在系统临时目录（如 `C:\Users\xxx\AppData\Local\Temp\tmpXXX`），不便于用户查看和管理。

**解决方案**：

- ✅ 统一保存到项目根目录下的 `generated_code/` 目录
- ✅ 文件名带时间戳，方便追踪历史记录
- ✅ 格式：`{analysis_type}_analysis_{YYYYMMDD_HHMMSS}.py`

**示例**：

### 2. 错误反馈循环机制（Error Feedback Loop）

**架构理念**：

用户期望的智能执行流程：

```
RunnerAgent 接收 task/config
    ↓
选择执行方式: hydromodel API  OR  生成代码执行
    ↓
执行并捕获错误
    ↓
错误分析 (error_type, root_cause, fix_suggestions)
    ↓
    ├─ 可修复错误 → 重新生成代码（附带错误信息）→ 重试
    │   ↓
    │   循环最多N次（默认3次）
    │   ↓
    └─ 不可修复错误 → 停止并返回详细错误报告
    ↓
成功 → 返回 result 给 DeveloperAgent
```

---

### 2.1 错误分析能力（`_analyze_execution_error`）

**功能**：自动识别错误类型并提供修复建议

**支持的错误类型**：

| 错误类型 | 检测关键词 | 根本原因 | 修复建议 | 是否可重试 |
|---------|-----------|---------|---------|-----------|
| `import_error` | ImportError, ModuleNotFoundError | 缺少Python库 | 安装缺失依赖 | ✅ 重新生成代码 |
| `data_error` | FileNotFoundError, no such file | 数据文件路径错误 | 检查路径、确认率定完成 | ⚠️ 需要更新config |
| `data_structure_error` | KeyError, IndexError | 数据结构不匹配 | 检查格式、更新代码 | ✅ 重新生成代码 |
| `runtime_error` | ValueError, TypeError | 数据类型/值错误 | 添加验证和转换 | ✅ 重新生成代码 |
| `general_runtime_error` | 其他 | 未预期的错误 | 检查堆栈信息 | ✅ 重新生成代码 |

**实现位置**：
- `hydroagent/agents/runner_agent.py`: Line 1744-1823

**示例输出**：
```python
{
    "error_type": "data_error",
    "root_cause": "数据文件路径不正确或文件不存在",
    "fix_suggestions": [
        "检查数据文件路径",
        "确认calibration已完成并生成了结果文件"
    ],
    "needs_config_update": True,
    "needs_code_regeneration": False
}
```

### 2.2 带反馈的代码生成（`_generate_code_with_feedback`）

**功能**：智能重试机制，自动从错误中学习并改进代码

**流程**：

```
1. 第1次尝试：使用原始prompt生成代码
   ↓
   执行 → 成功 ✅ 返回结果
   ↓
   执行 → 失败 ❌ 分析错误

2. 第2次尝试：使用增强prompt（包含上次错误信息）
   enhanced_params = {
       "previous_error": "FileNotFoundError: ...",
       "error_analysis": {错误分析结果},
       "retry_attempt": 2
   }
   ↓
   生成改进的代码 → 执行
   ↓
   成功 ✅ / 失败 ❌

3. 重复直到成功或达到max_retries（默认3次）
```

**实现位置**：
- `hydroagent/agents/runner_agent.py`: Line 1825-1933

**示例日志**：
```
[RunnerAgent] 开始代码生成（带错误反馈），最大重试3次
[RunnerAgent] 尝试 1/3
[RunnerAgent] 代码执行失败（尝试 1/3）
[RunnerAgent] 错误类型: data_error
[RunnerAgent] 错误可能可修复，准备重试...
[RunnerAgent] 尝试 2/3
[RunnerAgent] 代码执行成功（尝试 2/3）
```

### 2.3 集成到 `_run_custom_analysis`

**修改前**：
```python
# 简单的两步流程
code_result = self._generate_analysis_code(analysis_type, params)
exec_result = self._execute_generated_code(code_result["code_file"])
# 如果失败，直接返回错误
```

**修改后**：
```python
# 智能重试流程
result = self._generate_code_with_feedback(
    analysis_type=analysis_type,
    params=parameters,
    max_retries=3  # 可配置
)

if result["status"] == "success":
    # 返回成功结果 + 尝试次数 + 错误历史
    return {
        "status": "success",
        "attempts": 2,  # 尝试了2次
        "error_history": [...]  # 第1次的错误记录
    }
```

## 🔄 当前架构 vs 理想架构

### 当前实现（v4.0）

```
RunnerAgent (独立错误处理)
    ↓
生成代码 → 执行 → 错误分析 → 重新生成代码
    ↓ (内部循环，最多3次)
成功/失败 → 返回结果给 DeveloperAgent
```

**特点**：
- ✅ RunnerAgent内部实现错误反馈循环
- ✅ 自动分析错误类型并智能重试
- ✅ 记录完整的错误历史
- ⚠️ 目前仅限于代码重新生成，**尚未**与InterpreterAgent交互

---

### 理想架构（未来增强）

```
RunnerAgent → 执行失败
    ↓
错误分析：needs_config_update = True
    ↓
回调 InterpreterAgent
    ↓
InterpreterAgent 重新生成 config
    ↓
RunnerAgent 使用新 config 重新执行
    ↓
循环直到成功
```

**需要的改进**：
1. ⏳ RunnerAgent持有InterpreterAgent引用（或通过回调机制）
2. ⏳ InterpreterAgent支持基于错误信息重新生成config
3. ⏳ Pipeline级别的错误恢复机制

**为什么目前未实现**：
- 需要修改整个Pipeline的初始化逻辑
- 涉及多个Agent之间的通信协议
- 需要设计合理的回调接口
- 这是一个较大的架构改动，需要单独规划

## 📝 使用示例

### 基本使用

```python
from hydroagent.agents.runner_agent import RunnerAgent
from hydroagent.core.llm_interface import create_llm_interface

# 创建LLM接口
llm = create_llm_interface("openai", "qwen-turbo", api_key=API_KEY)
code_llm = create_llm_interface("openai", "qwen-coder-turbo", api_key=API_KEY)

# 创建RunnerAgent
runner = RunnerAgent(
    llm_interface=llm,
    code_llm_interface=code_llm,  # 🆕 代码专用LLM
    workspace_dir=Path("results/experiment_4")
)

# 执行自定义分析（自动带错误反馈）
result = runner._run_custom_analysis(
    config={...},
    parameters={
        "analysis_type": "runoff_coefficient",
        "basin_id": "01013500",
        "code_gen_max_retries": 5  # 🆕 可配置最大重试次数
    }
)

# 检查结果
if result["status"] == "success":
    print(f"✅ 成功（尝试{result['attempts']}次）")
    print(f"生成的文件: {result['generated_files']}")
    print(f"代码位置: {result['code_file']}")  # generated_code/...
else:
    print(f"❌ 失败（尝试{result['attempts']}次）")
    print(f"错误历史: {result['error_history']}")
```

## 🎯 核心改进亮点

### 1. 用户体验提升
- **代码可见性**：所有生成的代码集中在 `generated_code/` 目录，方便查看和学习
- **智能重试**：遇到错误自动尝试修复，减少人工干预
- **透明性**：完整记录错误历史和尝试次数

### 2. 系统鲁棒性提升
- **错误分类**：5种错误类型，针对性处理
- **自愈能力**：轻微错误（如数据路径）可自动修复
- **失败隔离**：不可恢复错误立即停止，避免资源浪费

### 3. 可扩展性
- **模块化设计**：错误分析和重试逻辑独立于具体任务
- **配置灵活**：`max_retries` 可根据任务复杂度调整
- **未来增强**：预留了与InterpreterAgent交互的接口

## ✅ 总结

**v4.0 实现了什么**：

1. ✅ **代码文件统一管理**：`generated_code/` 目录 + 时间戳文件名
2. ✅ **RunnerAgent内部错误反馈循环**：
   - 错误分析（5种类型）
   - 智能重试（最多N次）
   - 附带错误信息重新生成代码
3. ✅ **完整的错误历史记录**：便于调试和分析

**v4.0 尚未实现**：

1. ⏳ **跨智能体错误反馈**：RunnerAgent ↔ InterpreterAgent 交互
2. ⏳ **Config动态更新**：基于错误分析自动调整config
3. ⏳ **Pipeline级错误恢复**：整个流程的容错机制

**核心哲学**：

> **先实现单Agent的智能化，再推广到多Agent协作**
>
> v4.0先在RunnerAgent内部实现错误分析和自修复，验证机制有效性，然后在v5.0扩展到整个Pipeline的智能错误恢复。
